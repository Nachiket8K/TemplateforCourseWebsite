<h1>Assignments</h1>
<div class="article-style">
<br/>
<h2 id="overview">Overview</h2>
<p>There are 3 graded assignments in the course. These will assess the programming and reasoning skills developed in the weeks before the assignment submissions. The contents of the assignments will be released according to the <a href="/courses/epa1316/introduction/schedule/">schedule</a>.</p>
<ul>
<li><a href="../../assignments/assignment-01.zip">Assignment 01</a> - Data Collection and Wrangling (15%)</li>
<li><a href="../../assignments/assignment-02.zip">Assignment 02</a> - Geographic Visualisation (15%)</li>
<li><a href="../../assignments/assignment-03.zip">Assignment 03</a> - Prediction/Inference (20%)</li>
</ul>
<h3 id="marking-criteria-for-assignments">Marking Criteria for assignments</h3>
<p>The following rubric will be used for assessments. See the column with points (2) for what is expected of an ideal assignment. Points will be deducted if elements are missing and/or incorrectly done:</p>
<table>
<thead>
<tr>
<th>Criteria</th>
<th>Indicative weight</th>
<th>Needs Improvement (0.5-0.9)</th>
<th>Meets Some Expectations (1.0-1.9)</th>
<th>Exceeds Expectation (2)</th>
</tr>
</thead>
<tbody>
<tr>
<td>Output</td>
<td>0.5</td>
<td>the code does not produce the expected output</td>
<td>the code partially produces the expected output</td>
<td>the code shows the expected output<br/>- code runs<br/>- code uses reproducible paths<br/>- code has no errors or warnings<br/>- expected output is clearly presented through code/markdown explanations and figures</td>
</tr>
<tr>
<td>Formatting</td>
<td>0.5</td>
<td>code is not formatted at all or messy</td>
<td>code is partially formatted</td>
<td>code is properly formatted<br/>- variables and functions are named meaningfully like ‚Äúcrime_incidence as opposed to c_2513‚Äù.<br/>- code is written using functions<br/>- code is written in sections using headers and explanations in markdown<br/>- code is commented</td>
</tr>
<tr>
<td>Methods<br/>(e.g. Tidy Data, EDA, Graphical Excellence, Spatial Autocorrelation, Network Weights, Regression, etc.)</td>
<td>2</td>
<td>code/output shows no evidence of understanding of methods presented in class and homework exercises</td>
<td>code/output shows some evidence of understanding of methods presented in class and homework exercises</td>
<td>code/output illustrates clear evidence of understanding of methods presented in class and homework exercises <br/>- models are used correctly<br/>- use of visual inspection to explain model outputs<br/>- axes labels are named meaningfully, legends are present</td>
</tr>
<tr>
<td>Documentation (Markdown/Comments)</td>
<td>2</td>
<td>there is some documentation to follow your code. (e.g. why did you drop a certain variable or why did you choose one over another?)</td>
<td>there is some documentation explaining your code‚Äôs logic and your choices in the analysis but these choices or logic is not clear.</td>
<td>there is extensive documentation explaining your code‚Äôs logic and your choices in the analysis. <br/>- hypothesis clearly stated<br/>- explanation of model choices clearly presented (ex. variables, number of neighbours or clusters, etc.)<br/>- interpretation of model results, weakness and strengths (ex. errors in regression)<br/>- well substantiated argumentation of results and choices<br/>- plots are clearly described and interpreted</td>
</tr>
</tbody>
</table>
<h3 id="submission">Submission</h3>
<p>Submit assignments and final project on Brightspace. Formats and naming conventions are given within the respective files.</p>
